[PROJECT CONTEXT: aicli]
aicli is a shared LLM memory platform that allows Claude Code, the CLI, Cursor, and a web UI to access the same project context across tools. It uses a 3-layer memory architecture (raw captures → structured artifacts → approved work items), with multi-LLM async DAG pipelines, pgvector semantic search, and role-based agents defined in YAML templates. The app is currently focused on stabilizing pipeline execution, UI performance, and embedding consistency across 212+ work items.
Stack: language_cli=Python 3.12, cli_framework=prompt_toolkit + rich, backend_framework=FastAPI + uvicorn, backend_auth=JWT (python-jose + bcrypt) + DEV_MODE toggle, database=PostgreSQL 15+ with pgvector (1536-dim, text-embedding-3-small)
In progress: Node properties dynamic model dropdown: provider-model mappings cached; when provider changes, available models update via combobox, Execution logs per-role visibility: global 'Execution Log' toggle + per-stage <details> toggles showing input, tool calls (ReAct steps), observations, and output with duration/cost, Resizable execution panel: left-edge drag handle (7px col-resize zone), saves width to localStorage, min 300px / max 85% viewport
Decisions: Memory 3-layer architecture: raw captures (mem_mrr_*) → structured artifacts (mem_ai_project_facts) → approved work items (mem_work_items); ONLY approved items (UC/FE/BU/TA prefix) embed to pgvector; re-embedding on item change automatic via update() call to _embed_work_item(); Single source of truth: /memory POST endpoint is ONLY writer to project_state.json via get_project_context() + Haiku synthesis; CLAUDE.md, CODE.md, PROJECT.md all regenerated from single JSON state; Work item hierarchy: unified mem_work_items with wi_type (use_case/feature/bug/task/requirement), user_status TEXT (open/pending/in-progress/review/done), wi_parent_id linking children to use_case parents; wi_id progression: AI#### (draft) → UC/FE/BU/TA#### (approved)
