[PROJECT CONTEXT: aicli]
aicli is a unified LLM memory system that allows Claude Code, aicli CLI, Cursor, and a web UI to access the same project context across all AI sessions. It combines a 3-layer memory architecture (raw captures → structured facts → approved work items), YAML-defined multi-LLM pipelines with role-based agent execution, and pgvector semantic search over a PostgreSQL backend, with auto-deployment via git hooks to keep project memory synchronized.
Stack: language_cli=Python 3.12, cli_framework=prompt_toolkit + rich, backend_framework=FastAPI + uvicorn, backend_auth=JWT (python-jose + bcrypt) + DEV_MODE toggle, database=PostgreSQL 15+ with pgvector (1536-dim, text-embedding-3-small)
In progress: Work item classification pipeline: DELETE AI draft rows, classify new backlog items via /wi/{project}/classify endpoint, AI→UC/FE/BU/TA promotion workflow, Memory synthesis via /memory POST: project_state.json regeneration, Haiku-driven fact extraction to mem_ai_project_facts, document upload and ingestion, Auto-deploy hook integration: stop hook execution with auto_commit_push.sh, memory/work item sync back to central repo after Claude Code sessions
Decisions: Memory 3-layer architecture: raw captures (mem_mrr_* tables) → structured artifacts (mem_ai_project_facts via /memory POST + Haiku synthesis) → approved work items (mem_work_items with wi_parent_id hierarchy); ONLY approved items (UC/FE/BU/TA prefix) embed to pgvector; Single source of truth: /memory POST endpoint is ONLY writer to project_state.json via get_project_context() + Haiku synthesis; CLAUDE.md, CODE.md, PROJECT.md all regenerated from single JSON state; Work item hierarchy: unified mem_work_items with wi_type (use_case/feature/bug/task/requirement), user_status TEXT (open/pending/in-progress/review/done), wi_parent_id linking children to use_case parents; wi_id progression: AI#### (draft) → UC/FE/BU/TA#### (approved)
