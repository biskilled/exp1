## project_state.json (the hub)

Generated by `/memory` â†’ Haiku reads recent prompts + PROJECT.md â†’ writes JSON.
All context files are rendered FROM this file (no direct LLM during render).

```
Fields used by renderer:
  tech_stack       â†’ CLAUDE.md "Stack & Architecture"
  key_decisions    â†’ CLAUDE.md "Key Architectural Decisions"
  in_progress      â†’ CLAUDE.md "In Progress"
  project_summary  â†’ CLAUDE.md header (short description)
```

PROJECT.md is **authoritative**: sections Vision / Core Goals / Memory Architecture
feed the Haiku synthesis prompt. If PROJECT.md contradicts project_state.json,
PROJECT.md wins on next `/memory` run.

## Work Items â€” Gap Analysis

### Classification quality

| Aspect | State | Notes |
|--------|-------|-------|
| Use case grouping | âś“ | Haiku creates use_cases (item_level=3) then children linked via wi_parent_id |
| Item types | âś“ | use_case / feature / bug / task / policy / requirement â€” from YAML prompt |
| score_importance (0-5) | âś“ | AI-assigned: 0=trivial, 5=critical/blocking |
| score_status (0-5) | âś“ | AI completeness: 0=not started, 5=done |
| user_status (TEXT) | âś“ | User-managed: openâ†’pendingâ†’in-progressâ†’reviewâ†’done |
| User tags (phase/feature) | âś“ | Event tags flow into items via `_update_item_tags()`; classification prompt uses tag priority |

**Two parallel status fields** â€” `score_status` (int, AI assessment) and `user_status` (TEXT, user set) can diverge. No sync between them. By design: AI scores the completeness; user manages the workflow status.

### Approved items â€” what is well-defined

| Field | Populated on approve | Re-populated on update |
|-------|---------------------|----------------------|
| `wi_id` (US0001, FE0001â€¦) | âś“ | Never changes |
| `approved_at` | âś“ | Never changes |
| `embedding` VECTOR(1536) | âś“ (OpenAI) | âś“ **Now re-embeds on name/summary/deliveries change** |
| `mrr_ids` linked events | âś“ (mem_mrr_* rows tagged) | Not updated on edit |
| `acceptance_criteria` | Via `/wi pipeline` (PM agent) | Yes (pipeline re-run) |
| `implementation_plan` | Via `/wi pipeline` (Architect) | Yes (pipeline re-run) |
| use_case `.md` file | âś“ on use_case approve | On `/wi/{id}/md/refresh` |

### What appears in CLAUDE.md "Active Features"

Query: approved + not deleted + not completed + user_status NOT in (done, blocked).
Types: use_case, feature, bug, task â€” sorted bugs first, then by score_importance.
Shows: `wi_id`, `name`, `[status]`, `due_date`.
Does NOT show: summary, acceptance_criteria, implementation_plan.

### Conflict / consistency

| Scenario | Handled? |
|----------|----------|
| Child due_date > parent due_date | âś“ Blocked at update |
| UC due_date changed â†’ cascade to children | âś“ Auto-cascade |
| Re-parent to wrong type | âś“ Blocked (same type only) |
| Re-classify after edits (semantic drift) | âś— No â€” embedding stale until edit triggers re-embed |
| Two users edit same item concurrently | âś— No optimistic lock â€” last write wins |
| score_status vs user_status out of sync | By design â€” two separate fields |

## How New Requirements Override Old Ones

1. Edit `PROJECT.md` â€” update Memory Architecture / Vision / Core Goals
2. Run `/memory` â€” Haiku reads PROJECT.md as authoritative, replaces stale `key_decisions`
3. All context files (CLAUDE.md, cursorrules, etc.) re-rendered from updated `project_state.json`

Between `/memory` runs, CLAUDE.md reflects the last cached `project_state.json`.
